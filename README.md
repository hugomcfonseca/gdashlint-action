# gdashlint-action

GitHub Action for linting Grafana dashboard JSON files with [`gdashlint`](https://github.com/hugomcfonseca/gdashlint).

By default, the action downloads the requested `gdashlint` release binary from GitHub Releases and runs it with `--format github`, so findings are emitted as GitHub Actions annotations.

## Why the action defaults to release binaries

A Docker-based action is simple and reproducible, but it is not the best default for this project:

- It depends on Docker being available on the runner.
- It requires pulling an image for every cold run, which can be slower than downloading the small CLI archive.
- It can be affected by Docker Hub pull rate limits or registry availability.
- It does not work naturally on non-Linux runners.

The preferred approach is a composite action that downloads the official `gdashlint` archive from GitHub Releases. This avoids Docker Hub rate limits and keeps the action aligned with the same artifacts published by the CLI release process. Docker remains available through `distribution: docker` for users who explicitly prefer container execution.

## Usage

```yaml
name: Dashboard lint

on:
  pull_request:
  push:
    branches: [main]

permissions:
  contents: read

jobs:
  gdashlint:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@9f698171ed81b15d1823a05fc7211befd50c8ae0 # v6.0.3

      - name: Lint dashboards
        uses: hugomcfonseca/gdashlint-action@v0.2.0
        with:
          version: v0.2.0
          paths: dashboards/
```

## With a config file

```yaml
- name: Lint dashboards
  uses: hugomcfonseca/gdashlint-action@v0.2.0
  with:
    version: v0.2.0
    paths: dashboards/
    config: gdashlint.yaml
```

## Multiple paths

`paths` is newline-separated:

```yaml
- name: Lint dashboards
  uses: hugomcfonseca/gdashlint-action@v0.2.0
  with:
    version: v0.2.0
    paths: |
      dashboards/team-a
      dashboards/team-b
```

## Non-blocking annotations

Use `fail-on: none` to publish annotations without failing the workflow:

```yaml
- name: Lint dashboards
  uses: hugomcfonseca/gdashlint-action@v0.2.0
  with:
    version: v0.2.0
    paths: dashboards/
    fail-on: none
```

## Preview automatic fixes

`gdashlint fix --dry-run` previews safe remediations without writing files:

```yaml
- name: Preview dashboard fixes
  uses: hugomcfonseca/gdashlint-action@v0.2.0
  with:
    command: fix
    version: v0.2.0
    paths: dashboards/
    dry-run: true
```

To apply fixes in a workflow, use `command: fix` without `dry-run`. The action automatically passes `--yes` because GitHub-hosted runners do not provide interactive input.

## Automatically publish generated fixes

`gdashlint-action` intentionally stops at modifying files in the checked-out workspace. Committing, pushing, or opening pull requests is best handled by dedicated GitHub Actions so each workflow can choose its own review and branch policy.

The recommended approach is to open a pull request with the generated fixes:

```yaml
name: Fix dashboards

on:
  workflow_dispatch:

permissions:
  contents: write
  pull-requests: write

jobs:
  gdashlint-fix:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@9f698171ed81b15d1823a05fc7211befd50c8ae0 # v6.0.3

      - name: Apply gdashlint fixes
        uses: hugomcfonseca/gdashlint-action@v0.2.0
        with:
          command: fix
          version: v0.2.0
          paths: dashboards/

      - name: Create pull request
        uses: peter-evans/create-pull-request@5f6978faf089d4d20b00c7766989d076bb2fc7f1 # v8.1.1
        with:
          commit-message: "fix: apply gdashlint remediations"
          title: "fix: apply gdashlint remediations"
          body: |
            Automated dashboard fixes generated by gdashlint.
          branch: gdashlint/automated-fixes
```

If your repository policy allows direct commits to the current branch, you can instead compose with a commit action:

```yaml
name: Fix dashboards

on:
  workflow_dispatch:

permissions:
  contents: write

jobs:
  gdashlint-fix:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@9f698171ed81b15d1823a05fc7211befd50c8ae0 # v6.0.3

      - name: Apply gdashlint fixes
        uses: hugomcfonseca/gdashlint-action@v0.2.0
        with:
          command: fix
          version: v0.2.0
          paths: dashboards/

      - name: Commit gdashlint fixes
        uses: stefanzweifel/git-auto-commit-action@04702edda442b2e678b25b537cec683a1493fcb9 # v7.1.0
        with:
          commit_message: "fix: apply gdashlint remediations"
          file_pattern: dashboards/**/*.json
```

Prefer pull requests over direct pushes for protected branches, shared repositories, and workflows triggered by untrusted contributions. Avoid using `pull_request_target` to apply fixes to untrusted pull request code unless you have carefully reviewed the security implications.

## List available rules

```yaml
- name: List gdashlint rules
  uses: hugomcfonseca/gdashlint-action@v0.2.0
  with:
    command: rules
    version: v0.2.0
    format: text
```

## Docker execution

Docker execution is still supported for teams that want container isolation or already mirror the image to another registry:

```yaml
- name: Lint dashboards with Docker
  uses: hugomcfonseca/gdashlint-action@v0.2.0
  with:
    distribution: docker
    image: docker.io/hugomcfonseca/gdashlint
    version: v0.2.0
    paths: dashboards/
```

## Inputs

| Input          |                             Default | Description                                                                                   |
| -------------- | ----------------------------------: | --------------------------------------------------------------------------------------------- |
| `command`      |                              `lint` | `gdashlint` command to run: `lint`, `fix`, or `rules`.                                        |
| `paths`        |                                 `.` | Newline-separated dashboard files or directories to lint or fix. Ignored by `command: rules`. |
| `config`       |                               empty | Optional path to a `gdashlint` YAML configuration file.                                       |
| `format`       |                            `github` | Output format: `github`, `text`, or `json`.                                                   |
| `fail-on`      |                      config/default | Minimum severity that fails: `error`, `warning`, `info`, or `none`. Used by `lint` and `fix`. |
| `sort`         |                      config/default | Sort mode: `severity` or `file`. Used by `lint` and `fix`.                                    |
| `dry-run`      |                             `false` | Preview remediations without writing files. Used by `fix`.                                    |
| `mode`         |                          `in-place` | Fix write mode: `in-place` or `copy`. Used by `fix`.                                          |
| `suffix`       |                            `.fixed` | Suffix used before the extension in copy mode. Used by `fix`.                                 |
| `version`      |                            `latest` | `gdashlint` release tag to run. Pin this to a release tag for reproducibility.                |
| `distribution` |                            `binary` | `binary` downloads GitHub Release archives; `docker` runs the container image.                |
| `image`        | `docker.io/hugomcfonseca/gdashlint` | Container image repository to run when `distribution: docker`.                                |
| `github-token` |                      workflow token | Token used to resolve `version: latest`.                                                      |

## Version pinning

For reproducible workflows, pin both the action reference and CLI version. Pinning the action to a full commit SHA gives the strongest supply-chain protection; the version comment keeps the workflow readable:

```yaml
uses: hugomcfonseca/gdashlint-action@<release-commit-sha> # v0.2.0
with:
  version: v0.2.0
```

The examples above use the release tag for `gdashlint-action` because this repository's release commit SHA is only known after the release is published. Third-party actions in the examples are pinned to full commit SHAs with version comments.

Using `version: latest` is convenient while experimenting, but it can change behavior when new `gdashlint` releases are published. Published workflow examples intentionally pin the `gdashlint` version.

## Requirements

- `distribution: binary` supports GitHub-hosted Linux and macOS runners for `amd64` and `arm64` release assets.
- `distribution: binary` verifies downloaded release archives against the `checksums.txt` file published with each `gdashlint` release.
- `distribution: docker` requires a runner with Docker available. GitHub-hosted `ubuntu-latest` runners satisfy this requirement.
